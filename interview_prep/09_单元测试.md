## 1. 测试基础

### 1.1 为什么要写单元测试

| 价值 | 说明 |
|------|------|
| **防止回归** | 修改代码后快速发现是否破坏了已有功能 |
| **文档行为** | 测试 = 可执行的需求文档，描述代码应该做什么 |
| **设计反馈** | 难测试 = 代码耦合度高，测试驱动更好的设计 |
| **信心保障** | 重构时有安全网，放心改 |
| **加速开发** | 减少手动验证时间，CI 自动跑 |
| **降低 bug 修复成本** | Bug 发现越早，修复成本越低 |

### 1.2 测试金字塔

```
          ╱‾‾‾‾‾╲
         ╱  E2E   ╲           少量，慢，昂贵
        ╱ (端到端)  ╲         验证完整流程
       ╱─────────────╲
      ╱  Integration  ╲      适量，中等速度
     ╱    (集成测试)    ╲     验证模块协作
    ╱───────────────────╲
   ╱     Unit Tests       ╲   大量，快速，廉价
  ╱       (单元测试)        ╲  验证单个函数/类
 ╱───────────────────────────╲
```

**面试金句**："测试金字塔的核心思想是：底层（单元测试）应该最厚最多，越上层越少。单元测试快且稳定，E2E 测试慢且脆弱，所以大多数测试应该在单元层。"

### 1.3 好的测试应该具备 F.I.R.S.T

| 特性 | 说明 |
|------|------|
| **Fast (快速)** | 一个测试在毫秒级完成，整个 suite 在秒级完成 |
| **Independent (独立)** | 测试之间不依赖，可以任意顺序运行 |
| **Repeatable (可重复)** | 每次运行结果一致，不依赖外部环境 |
| **Self-validating (自验证)** | 自动判断通过/失败，不需要人工检查 |
| **Timely (及时)** | 在代码之前（TDD）或与代码同时编写 |

### 2.2 基本测试结构

```cpp
#include <gtest/gtest.h>

// TEST(TestSuiteName, TestName)

    EXPECT_EQ(Add(5, 0), 5);
```

## 3. 断言 (Assertions)

### 3.1 ASSERT vs EXPECT 区别

| 宏 | 失败时行为 | 使用场景 |
|----|-----------|---------|
| `EXPECT_*` | 记录失败，继续执行 | 大部分场景（获取更多失败信息） |
| `ASSERT_*` | 记录失败，**立即返回** | 后续代码依赖此断言成功（如指针非空才能解引用） |

### 3.2 显式布尔断言

```cpp
TEST(AssertionTest, ExplicitBoolean) {
    bool condition = true;

    ASSERT_TRUE(condition);   // 断言为真
    EXPECT_TRUE(condition);

    ASSERT_FALSE(false);      // 断言为假
    EXPECT_FALSE(false);
}
```

### 3.3 比较断言

```cpp
TEST(AssertionTest, ComparisonAssertions) {
    // EXPECT_EQ(expected, actual) — 相等
    EXPECT_EQ(4, 2 + 2);
    EXPECT_EQ(std::string("hello"), std::string("hello"));

    // EXPECT_NE — 不等
    EXPECT_NE(5, 2 + 2);

    // EXPECT_LT — 小于
    EXPECT_LT(3, 5);

    // EXPECT_LE — 小于等于
    EXPECT_LE(5, 5);
    EXPECT_LE(4, 5);

    // EXPECT_GT — 大于
    EXPECT_GT(5, 3);

    // EXPECT_GE — 大于等于
    EXPECT_GE(5, 5);
    EXPECT_GE(6, 5);
}
```

### 3.4 字符串断言

```cpp
#include <gtest/gtest.h>
#include <string>

TEST(AssertionTest, StringAssertions) {
    std::string str1 = "hello";
    std::string str2 = "hello";

    // C++ string 用 EXPECT_EQ
    EXPECT_EQ(str1, str2);

    // C 风格字符串用 EXPECT_STREQ（比较内容，非指针）
    const char* cstr1 = "world";
    const char* cstr2 = "world";
    EXPECT_STREQ(cstr1, cstr2);   // 比较内容
    EXPECT_STRNE(cstr1, "hello"); // 内容不等

    // EXPECT_STRCASEEQ — 忽略大小写比较
    EXPECT_STRCASEEQ("HELLO", "hello");

    // 字符串包含检查
    EXPECT_STRNE(str1.c_str(), nullptr);  // 不是 nullptr
}
```

### 3.5 浮点数断言

```cpp
#include <cmath>

TEST(AssertionTest, FloatingPoint) {
    double a = 0.1 + 0.2;
    double b = 0.3;

    // 不要用 EXPECT_EQ 比较浮点数！
    // EXPECT_EQ(a, b);  // 可能失败：0.1+0.2 != 0.3 in floating point

    // 使用 EXPECT_NEAR(expected, actual, tolerance)
    EXPECT_NEAR(a, b, 1e-9);

    // EXPECT_DOUBLE_EQ — 默认 4 ULPs 容差
    EXPECT_DOUBLE_EQ(0.3, 0.1 + 0.2);

    EXPECT_FLOAT_EQ(0.3f, 0.1f + 0.2f);
}
```

### 3.6 异常断言

```cpp
#include <stdexcept>

int Divide(int a, int b) {
    if (b == 0) throw std::invalid_argument("division by zero");
    return a / b;
}

void MightThrow(bool shouldThrow) {
    if (shouldThrow) throw std::runtime_error("bad");
}

TEST(AssertionTest, ExceptionAssertions) {
    // EXPECT_THROW — 期望抛出指定类型异常
    EXPECT_THROW(Divide(10, 0), std::invalid_argument);

    // EXPECT_NO_THROW — 期望不抛出异常
    EXPECT_NO_THROW(Divide(10, 2));

    // EXPECT_ANY_THROW — 期望抛出任意异常
    EXPECT_ANY_THROW(MightThrow(true));
    EXPECT_NO_THROW(MightThrow(false));

    // 检查异常消息
    EXPECT_THROW({
        try {
            Divide(10, 0);
        } catch (const std::invalid_argument& e) {
            EXPECT_STREQ("division by zero", e.what());
            throw;  // 重新抛出让 EXPECT_THROW 捕获
        }
    }, std::invalid_argument);
}
```

### 3.7 死亡测试 (Death Tests)

```cpp
#include <cassert>

void DieIfNegative(int x) {
    assert(x >= 0);  // 违反时 abort()
}

TEST(DeathTest, ShouldAbort_WhenNegativeInput) {
    // ASSERT_DEATH — 期望以某种方式终止
    ASSERT_DEATH(DieIfNegative(-1), "");

    // ASSERT_DEATH_IF_SUPPORTED — 在不支持的平台不失败
    ASSERT_DEATH_IF_SUPPORTED(DieIfNegative(-1), "");

    // 检查退出码
    ASSERT_EXIT(exit(42), ::testing::ExitedWithCode(42), "");
}
```

### 3.8 断言完整速查表

```cpp
// ┌────────────────────┬────────────────────────────────┐
// │ 断言宏              │ 用途                           │
// ├────────────────────┼────────────────────────────────┤
// │ ASSERT_TRUE(c)     │ c 为 true                       │
// │ ASSERT_FALSE(c)    │ c 为 false                      │
// │ ASSERT_EQ(a,b)     │ a == b (支持 << 自定义输出)      │
// │ ASSERT_NE(a,b)     │ a != b                          │
// │ ASSERT_LT(a,b)     │ a < b                           │
// │ ASSERT_LE(a,b)     │ a <= b                          │
// │ ASSERT_GT(a,b)     │ a > b                           │
// │ ASSERT_GE(a,b)     │ a >= b                          │
// │ ASSERT_STREQ(a,b)  │ C 字符串内容相等                 │
// │ ASSERT_STRNE(a,b)  │ C 字符串内容不等                 │
// │ ASSERT_STRCASEEQ   │ C 字符串忽略大小写相等            │
// │ ASSERT_FLOAT_EQ    │ float 相等 (4 ULPs)              │
// │ ASSERT_DOUBLE_EQ   │ double 相等 (4 ULPs)             │
// │ ASSERT_NEAR(a,b,t) │ |a-b| <= t                      │
// │ ASSERT_THROW(s,e)  │ s 抛出 e 类型异常                 │
// │ ASSERT_NO_THROW(s) │ s 不抛出异常                     │
// │ ASSERT_ANY_THROW(s)│ s 抛出任意异常                   │
// │ ASSERT_DEATH(s,r)  │ s 以匹配 r 的方式终止             │
// ├────────────────────┼────────────────────────────────┤
// │ EXPECT_*           │ 同上，失败后继续执行              │
// └────────────────────┴────────────────────────────────┘
```

### 3.9 自定义失败消息

```cpp
    EXPECT_EQ(actual, expected) << "Value mismatch at index "
```

## 4. Test Fixture (测试夹具)

### 4.1 基本 Fixture：SetUp / TearDown

```cpp
#include <gtest/gtest.h>
#include <vector>
#include <memory>

// 被测类
class Stack {
public:
    void Push(int value) { data_.push_back(value); }
    int Pop() {
        if (data_.empty()) throw std::out_of_range("empty");
        int val = data_.back();
        data_.pop_back();
        return val;
    }
    int Top() const {
        if (data_.empty()) throw std::out_of_range("empty");
        return data_.back();
    }
    size_t Size() const { return data_.size(); }
    bool IsEmpty() const { return data_.empty(); }
private:
    std::vector<int> data_;
};

// Fixture: 每个 TEST_F 前调用 SetUp，后调用 TearDown
class StackTest : public ::testing::Test {
protected:
    // SetUp — 每个测试前执行
    void SetUp() override {
        stack_ = std::make_unique<Stack>();
        // 预填充一些数据
        stack_->Push(1);
        stack_->Push(2);
        stack_->Push(3);
    }

    // TearDown — 每个测试后执行
    void TearDown() override {
        // 清理工作（unique_ptr 自动释放，这里作为示范）
        stack_.reset();
    }

    // 辅助方法 — 供测试用例使用
    void PushMany(size_t count) {
        for (size_t i = 0; i < count; ++i) {
            stack_->Push(static_cast<int>(i + 10));
        }
    }

    std::unique_ptr<Stack> stack_;
};

// 使用 TEST_F 而非 TEST
TEST_F(StackTest, ShouldHaveThreeElements_AfterSetUp) {
    EXPECT_EQ(stack_->Size(), 3);
    EXPECT_EQ(stack_->Top(), 3);
}

TEST_F(StackTest, ShouldReturnTop_WhenPop) {
    int val = stack_->Pop();
    EXPECT_EQ(val, 3);
    EXPECT_EQ(stack_->Size(), 2);
    EXPECT_EQ(stack_->Top(), 2);
}

TEST_F(StackTest, ShouldThrow_WhenPopEmpty) {
    stack_->Pop();  // 3
    stack_->Pop();  // 2
    stack_->Pop();  // 1
    EXPECT_TRUE(stack_->IsEmpty());
    EXPECT_THROW(stack_->Pop(), std::out_of_range);
}

TEST_F(StackTest, ShouldIncreaseSize_WhenPush) {
    size_t oldSize = stack_->Size();
    stack_->Push(42);
    EXPECT_EQ(stack_->Size(), oldSize + 1);
    EXPECT_EQ(stack_->Top(), 42);
}

TEST_F(StackTest, ShouldBeEmpty_WhenNewStack) {
    Stack emptyStack;
    EXPECT_TRUE(emptyStack.IsEmpty());
    EXPECT_EQ(emptyStack.Size(), 0);
}
```

### 4.2 静态 Fixture：SetUpTestCase / TearDownTestCase

```cpp
// 静态 Fixture — 整个 test suite 共享的昂贵资源
class DatabaseTest : public ::testing::Test {
protected:
    // 所有测试开始前执行一次 (注意是 static)
    static void SetUpTestSuite() {
        // 启动内存数据库（耗时操作，只做一次）
        db_ = new MockDatabase();
        db_->Connect();
        db_->LoadSchema();

        std::cout << "[SUITE] Database initialized once" << std::endl;
    }

    // 所有测试结束后执行一次 (注意是 static)
    static void TearDownTestSuite() {
        db_->Disconnect();
        delete db_;
        db_ = nullptr;

        std::cout << "[SUITE] Database cleaned up" << std::endl;
    }

    // 每个测试前后仍可用 SetUp / TearDown
    void SetUp() override {
        // 每个测试前开启事务
        db_->BeginTransaction();
    }

    void TearDown() override {
        // 每个测试后回滚，保证测试隔离
        db_->RollbackTransaction();
    }

    static MockDatabase* db_;
};

MockDatabase* DatabaseTest::db_ = nullptr;

TEST_F(DatabaseTest, ShouldInsertRecord_WhenValidData) {
    bool ok = db_->Insert("users", "name=Alice");
    EXPECT_TRUE(ok);
}

TEST_F(DatabaseTest, ShouldReturnError_WhenDuplicateKey) {
    db_->Insert("users", "id=1,name=Alice");
    bool ok = db_->Insert("users", "id=1,name=Bob");
    EXPECT_FALSE(ok);
}
```

## 5. Google Mock

### 5.1 Mock 基础概念

Mock 用于模拟外部依赖，使单元测试真正"单元"化。

```cpp
#include <gmock/gmock.h>

// 被测代码依赖的接口
class IFileSystem {
public:
    virtual ~IFileSystem() = default;
    virtual bool FileExists(const std::string& path) const = 0;
    virtual std::string ReadFile(const std::string& path) = 0;
    virtual void WriteFile(const std::string& path,
                           const std::string& content) = 0;
    virtual std::vector<std::string> ListFiles(
        const std::string& directory) = 0;
};

// 实际的实现（生产环境）
class RealFileSystem : public IFileSystem {
public:
    bool FileExists(const std::string& path) const override {
        return std::filesystem::exists(path);
    }
    std::string ReadFile(const std::string& path) override {
        std::ifstream in(path);
        return std::string(std::istreambuf_iterator<char>(in),
                           std::istreambuf_iterator<char>());
    }
    void WriteFile(const std::string& path,
                   const std::string& content) override {
        std::ofstream out(path);
        out << content;
    }
    std::vector<std::string> ListFiles(
        const std::string& directory) override {
        std::vector<std::string> result;
        for (const auto& entry :
             std::filesystem::directory_iterator(directory)) {
            result.push_back(entry.path().filename().string());
        }
        return result;
    }
};

// Mock 实现（测试环境）
class MockFileSystem : public IFileSystem {
public:
    // MOCK_METHOD(返回类型, 函数名, (参数列表), (修饰符))
    MOCK_METHOD(bool, FileExists, (const std::string& path),
                (const, override));
    MOCK_METHOD(std::string, ReadFile, (const std::string& path),
                (override));
    MOCK_METHOD(void, WriteFile,
                (const std::string& path, const std::string& content),
                (override));
    MOCK_METHOD(std::vector<std::string>, ListFiles,
                (const std::string& directory), (override));
};
```

### 5.2 EXPECT_CALL 详解

```cpp
using ::testing::Return;
using ::testing::_;
using ::testing::AtLeast;
using ::testing::Exactly;
using ::testing::AtMost;
using ::testing::Times;
using ::testing::WillOnce;
using ::testing::WillRepeatedly;

TEST(MockTest, BasicExpectCall) {
    MockFileSystem mock;

    // 期望 FileExists 被调用至少 1 次，参数任意，返回 true
    EXPECT_CALL(mock, FileExists(_))
        .Times(AtLeast(1))
        .WillRepeatedly(Return(true));

    // 期望 ReadFile("config.json") 被精确调用 1 次，返回特定内容
    EXPECT_CALL(mock, ReadFile("config.json"))
        .Times(Exactly(1))
        .WillOnce(Return("{\"port\": 8080}"));

    // 期望 WriteFile 被调用，用 WillOnce 返回不同值
    EXPECT_CALL(mock, ReadFile("data.txt"))
        .WillOnce(Return("data1"))
        .WillOnce(Return("data2"));

    // 使用 mock
    EXPECT_TRUE(mock.FileExists("anything.txt"));
    EXPECT_EQ(mock.ReadFile("config.json"), "{\"port\": 8080}");
    EXPECT_EQ(mock.ReadFile("data.txt"), "data1");
    EXPECT_EQ(mock.ReadFile("data.txt"), "data2");
}
```

### 5.3 ON_CALL vs EXPECT_CALL

```cpp
// ON_CALL: 定义默认行为（不关心调用次数）
// EXPECT_CALL: 定义期望行为（关心调用情况）

TEST(MockTest, OnCallVsExpectCall) {
    MockFileSystem mock;

    // ON_CALL — 设置默认行为（stub），不强制必须调用
    ON_CALL(mock, FileExists(_)).WillByDefault(Return(true));

    // EXPECT_CALL — 期望被调用，如果没调用测试失败
    EXPECT_CALL(mock, ReadFile("test.txt"))
        .Times(1)
        .WillOnce(Return("content"));

    // 不调用也 OK（没有 EXPECT_CALL 对 FileExists 的约束）
    mock.ReadFile("test.txt");  // 必须调用，否则失败
}
```

### 5.4 完整的 Mock 示例：数据库接口

```cpp
#include <gmock/gmock.h>

// ---------- 接口 ----------
struct User {
    int id;
    std::string name;
    std::string email;
};

class IUserRepository {
public:
    virtual ~IUserRepository() = default;
    virtual std::optional<User> FindById(int id) = 0;
    virtual std::vector<User> FindAll() = 0;
    virtual bool Save(const User& user) = 0;
    virtual bool Delete(int id) = 0;
};

// ---------- Mock ----------
class MockUserRepository : public IUserRepository {
public:
    MOCK_METHOD(std::optional<User>, FindById, (int id), (override));
    MOCK_METHOD(std::vector<User>, FindAll, (), (override));
    MOCK_METHOD(bool, Save, (const User& user), (override));
    MOCK_METHOD(bool, Delete, (int id), (override));
};

// ---------- 被测类 ----------
class UserService {
public:
    explicit UserService(std::shared_ptr<IUserRepository> repo)
        : repo_(std::move(repo)) {}

    std::string GetUserDisplayName(int id) {
        auto user = repo_->FindById(id);
        if (!user.has_value()) return "Unknown User";
        return user->name + " <" + user->email + ">";
    }

    bool CreateUser(const std::string& name, const std::string& email) {
        if (name.empty() || email.empty()) return false;
        User user{0, name, email};
        return repo_->Save(user);
    }

    bool TransferUser(int fromId, int toId) {
        auto from = repo_->FindById(fromId);
        auto to = repo_->FindById(toId);
        if (!from.has_value() || !to.has_value()) return false;
        // 合并逻辑 ...
        return repo_->Save(from.value()) && repo_->Save(to.value());
    }

private:
    std::shared_ptr<IUserRepository> repo_;
};

// ---------- 测试 ----------
class UserServiceTest : public ::testing::Test {
protected:
    void SetUp() override {
        mockRepo_ = std::make_shared<MockUserRepository>();
        service_ = std::make_unique<UserService>(mockRepo_);
    }

    std::shared_ptr<MockUserRepository> mockRepo_;
    std::unique_ptr<UserService> service_;
};

TEST_F(UserServiceTest, ShouldReturnFormattedName_WhenUserExists) {
    User alice{1, "Alice", "alice@example.com"};

    EXPECT_CALL(*mockRepo_, FindById(1))
        .WillOnce(Return(alice));

    auto result = service_->GetUserDisplayName(1);
    EXPECT_EQ(result, "Alice <alice@example.com>");
}

TEST_F(UserServiceTest, ShouldReturnUnknown_WhenUserNotFound) {
    EXPECT_CALL(*mockRepo_, FindById(999))
        .WillOnce(Return(std::nullopt));

    auto result = service_->GetUserDisplayName(999);
    EXPECT_EQ(result, "Unknown User");
}

TEST_F(UserServiceTest, ShouldReturnFalse_WhenCreateWithEmptyName) {
    // Save 不应该被调用
    EXPECT_CALL(*mockRepo_, Save(::testing::_)).Times(0);

    bool result = service_->CreateUser("", "alice@example.com");
    EXPECT_FALSE(result);
}

TEST_F(UserServiceTest, ShouldSaveUser_WhenCreateWithValidData) {
    EXPECT_CALL(*mockRepo_, Save(::testing::_))
        .WillOnce(Return(true));

    bool result = service_->CreateUser("Alice", "alice@example.com");
    EXPECT_TRUE(result);
}

TEST_F(UserServiceTest, ShouldCallSaveTwice_WhenTransferUser) {
    User from{1, "Alice", "a@e.com"};
    User to{2, "Bob", "b@e.com"};

    EXPECT_CALL(*mockRepo_, FindById(1)).WillOnce(Return(from));
    EXPECT_CALL(*mockRepo_, FindById(2)).WillOnce(Return(to));
```

## 7. 覆盖率

### 7.1 什么是代码覆盖率

覆盖率衡量测试执行了多少代码。提供量化指标，但不等于质量。

| 指标 | 含义 | 示例 |
|------|------|------|
| **行覆盖率** | 执行了多少行代码 | if (x>0) a(); b(); — x>0 时覆盖 100% |
| **分支覆盖率** | 执行了多少分支 | 上面例子中 else 分支未覆盖 → 50% |
| **函数覆盖率** | 调用了多少函数 | 10 个函数调用了 8 个 → 80% |
| **条件覆盖率** | 每个 boolean 子条件的 true/false | if (a && b) — 需要 a=T/F, b=T/F |
| **MC/DC** | Modified Condition/Decision Coverage | 安全关键系统要求，每个条件独立影响结果 |

```
示例：代码覆盖率和分支覆盖率的区别

void f(int x) {              行覆盖   分支覆盖
    if (x > 0) {              ✓        ✓ (true 分支)
        doA();                ✓
    } else {                  ✗        ✗ (false 分支)
        doB();                ✗
    }
}
测试: f(1)  → 行覆盖 60%, 分支覆盖 50%
```

### 7.2 gcov / lcov 使用

```bash
# 1. 编译时添加覆盖率标志
g++ -fprofile-arcs -ftest-coverage -g -O0 \
    -o my_test my_test.cpp -lgtest -lgtest_main -pthread
# 或用 CMake
# target_compile_options(my_lib PRIVATE --coverage)
# target_link_options(my_test PRIVATE --coverage)

# 2. 运行测试
./my_test

# 3. 生成覆盖率报告
gcov my_source.cpp                    # 生成 my_source.cpp.gcov 文本文件

# 4. 可视化 (lcov + genhtml)
lcov --capture --directory . \
     --output-file coverage.info
lcov --remove coverage.info \
     '/usr/*' '*/tests/*' '*/gtest/*' \
     --output-file coverage_filtered.info
genhtml coverage_filtered.info \
        --output-directory coverage_report

# 5. 查看报告
# 打开 coverage_report/index.html

# 精确到分支覆盖率
lcov --rc lcov_branch_coverage=1 \
     --capture --directory . --output-file coverage.info
genhtml coverage.info --output-directory coverage_report \
    --branch-coverage
```

### 7.3 CMake 集成覆盖率

```cmake
# CMakeLists.txt 中添加覆盖率选项
option(ENABLE_COVERAGE "Enable code coverage" OFF)

if(ENABLE_COVERAGE)
    if(CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")
        add_compile_options(--coverage)
        add_link_options(--coverage)
    endif()
endif()

# 添加覆盖率目标
add_custom_target(coverage
    COMMAND ${CMAKE_CTEST_COMMAND}
    COMMAND lcov --capture --directory . --output-file coverage.info
    COMMAND lcov --remove coverage.info
            '*/tests/*' '*/third_party/*' '/usr/*'
            --output-file coverage_filtered.info
    COMMAND genhtml coverage_filtered.info
            --output-directory coverage_report
    WORKING_DIRECTORY ${CMAKE_BINARY_DIR}
    COMMENT "Generating coverage report"
)
```

### 7.4 覆盖率目标

- **目标覆盖率**：通常 80%+ 行覆盖率是合理目标
- **不要迷信 100%**：100% 覆盖率 != 没有 Bug
- **关注未覆盖代码**：未覆盖的代码是测试盲区
- **新代码**：新功能应该接近 90%+
- **关键路径**：支付、安全相关必须 100%
- **防御性代码**：`default: assert(false)` 之类的可能很难覆盖

## 9. 测试最佳实践

### 9.1 AAA 模式 (Arrange-Act-Assert)

```cpp
// Arrange-Act-Assert: 每个测试遵循三段式结构
TEST(OrderServiceTest, ShouldApplyDiscount_WhenVipCustomer) {
    // Arrange (准备): 创建对象，设置前置条件
    auto mockRepo = std::make_shared<MockOrderRepository>();
    auto mockNotifier = std::make_shared<MockNotifier>();
    OrderService service(mockRepo, mockNotifier);

    Order order{1, 100.0, true};  // VIP 订单
    EXPECT_CALL(*mockRepo, Save(order)).WillOnce(Return(true));

    // Act (执行): 调用被测方法
    bool result = service.ProcessOrder(order);

    // Assert (断言): 验证结果
    EXPECT_TRUE(result);
    // 注：Mock 的 EXPECT_CALL 自动验证调用
}
```

### 9.2 测试命名

### 9.7 什么应该测试/什么不应该测试

| 应该测试 | 不应该测试 |

### 面试重点

2. **FIRST 原则**：什么是好的单元测试
3. **测试金字塔**：Unit > Integration > E2E 的比例和原因
4. **Mock vs Stub**：Mock 关心行为（是否调用），Stub 只返回预设值
5. **依赖注入与可测试性**：代码难测试 → 设计有问题
6. **覆盖率**：行覆盖、分支覆盖的区别，覆盖率不是银弹
7. **TDD 红-绿-重构**：先写测试，再写代码

### 高频考点

- **什么时候不写单元测试**：探索性代码、生命周期短的脚本、原型
- **好的测试命名**：`Should_xxx_When_xxx` 格式
- **EXPECT vs ASSERT**：失败后继续 vs 立即停止
- **SetUp vs SetUpTestCase**：每个测试 vs 整个 suite
- **代码覆盖率 100% 有意义吗**：没有！可以通过没有断言的测试达到 100%
- **私有方法要不要测**：通过公共接口间接测试，不应该直接测私有方法
- **如何测试多线程代码**：困难但可以通过模拟时间、注入调度器等方法测试
- **GMock 的 EXPECT_CALL vs ON_CALL**：期望 vs 默认行为

### 常见陷阱

1. **不写测试** — "代码太简单不需要测试" → 简单代码最容易被无意识地破坏
2. **测试实现而非行为** — 重构内部实现时测试全红
3. **测试间有依赖** — 全局状态、单例导致测试顺序敏感
4. **慢测试** — 单元测试应该毫秒级，集成测试才配秒级
5. **不覆盖边界/异常** — 只测 Happy Path
6. **Mock 太多** — Mock 太多说明类职责过多（SRP 问题）
7. **断言不足** — 有 Mock 但没有 ASSERT，测试永远绿
8. **测试代码质量差** — 测试也是产品代码，也需要可读性和 DRY
9. **忽略 Flaky Test** — "偶尔失败没事" → 习以为常后真失败看不出来
10. **覆盖率为指标考核** — 逆激励：开发者为了覆盖率写无用测试

### 建议背诵内容

1. **FIRST 原则**："Fast, Independent, Repeatable, Self-validating, Timely"

2. **AAA 三段式**："Arrange (准备) → Act (执行) → Assert (验证)"

3. **测试金字塔**："单元测试多而快，集成测试少而适中，E2E 更少而慢"

4. **TDD 三步**："红 — 写失败测试，绿 — 最小实现，重构 — 改善结构"

5. **Mock 核心三行**：
```cpp
class MockX : public IX {
    MOCK_METHOD(RetType, Method, (Args), (override));
};
EXPECT_CALL(mock, Method(_)).Times(1).WillOnce(Return(value));
```

6. **覆盖率的真相**："高覆盖率不等于高质量，低覆盖率一定意味着测试不足"

7. **测试命名黄金法则**："Should_ExpectedBehavior_When_Condition"

8. **依赖注入金句**："如果你的类很难测试，说明它的设计有问题。测试困难是坏设计的检测器。"

---

> **面试建议**：单元测试面试题往往是"你平时写单元测试吗？怎么写？"这种开放性问题。用具体的代码示例（AAA、Mock、Fixture）和实际的覆盖率数字来回答，远比背诵概念有说服力。准备一个真实的测试案例，从被测代码到测试代码完整讲述。
